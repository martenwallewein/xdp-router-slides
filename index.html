XDP, AF_XDP, eBPF and the SCION Border Router

+ Short HSR Deployment Update

Marten Gartner, Lars-Christian Schulz, David Hausheer

Motivation

  • Upstream Border Router (OSR) is slow.
  • We have the High Speed Router (HSR), but
    • Really difficult to deploy, limited in Hardware, topology changes required
    • No updates of changes, no chance to look in the source
    • Also not all features of SCIONLab covered
  • Therefor: We think about to do an add-on to Border Router to speed up certain data paths

XDP Border Router Add-on

  • Filter for SCION/UDP packets with path type SCION
  • There are forwarded in the kernel, the rest is passed to the OSR

What Happens in "Forwarding"

  • Increment CurHF (and in case of segment change CurInf)
  • Update SegId field in CurInf, verify HF
  • Send to nexthop or host in local AS

How to get SpeedUp?

  • Bypass Linux kernel using XDP entrypoint
  • eBPF provides the hook to XDP and runs code in the kernel
  • This code can write in packets and return an action (DROP, PASS, TX, etc)
  • This is fast:
    • Drop 26M pps on a single core ()
    • Forward 28M pps with 5 cores ()
  • So: We use XDP, we're done, it's fast... except for:
    • The eBPF programs attached to XDP need to be very short and lightweight to be fast for a single flow...

Single Flow - Single Core Problem

  • TOOD: Figure! A queue is mostly bound to a CPU
  • This means each packet arriving at the queue is handled by the CPU
  • The NIC normally decides to which queue a packet is sent by a 5 tuple (flow)
    • - src/dest addr and src/dest port, L4 proto
  • So a single flow (forwarding between two Border Routers) always ends up at the same CPU
  • Meaning: No parallelism for a single flow
  • Fun fact: This is exactly the same point why the HSR of Anapaya is limited

With AES, forwarding is not "lightweight"

  • With AES not using asm intrinsics, there is a lot of work to do to verify hop fields
  • So what can we do:
    • - RSS: Receive Side Scaling
      • - We change the 5 tuple to distribute packets over multiple flows (limited to particular NICs)
      • - We add source port randomization in the routers, so that we have different 5 tuples (this is open at anapaya since months...)
    • - We check this thing called AF_XDP

Receiver side: Packet rewriting

  • BPF app opens packets containing the UDP TargetPort of the dispatcher
  • Lookup L4 SCION UDP dest, write into UDP TargetPort

New SCION Data Flow

  • Only Bittorrent specific SCION packets are changed
  • Remaining SCION traffic remains untouched

First Results: Raw SCION/UDP

  • Measurements done on 10Gbit/s link with HSR, Jumbo Frames and simple bandwidth tester

Next Steps:

  • Fix SCION/QUIC packet sending (handshake error)
  • Test with Multipath Bittorrent over SCION
  • Continue with more experiments
  • Investigate why single connections are limited to max 2Gbit/s
  • Support multiple registered apps instead of hard coded port range
  • Aim for bringing this prototype upstream once its a working application

What does the Dispatcher?

  • Handles incoming and outgoing SCION packets on SCION hosts, does service resolution
  • From design perspective very useful
    • Last border router on path does not need to lokup l4 port
    • Dispatcher keeps map of registered apps/services, redirects packets
  • From performance perspective questionable
    • All SCION data packets traverse the kernel 2x on each host
